# BeeGFS 7.3.2 冷态性能基线测试

> 日期：2026-07-03 | 集群：4 节点 (157/150/151/152)
> 脚本：`tests/bench-full.sh` | 参数对齐 JuiceFS+Ceph cold-cache0

## 测试目的

在 BeeGFS 7.3.2 集群上测量冷态性能基线（写→清缓存→读），并与另一集群上的 JuiceFS+Ceph 方案做对比。

## 测试环境

| 项目 | 配置 |
|------|------|
| BeeGFS 版本 | 7.3.2 |
| 架构 | 1 mgmtd + 4 meta + 3 storage (6 targets) |
| 镜像 | metadata 2 buddy groups + storage 3 buddy groups |
| Stripe | Buddy Mirror, 1M chunk, 3 targets |
| 客户端 | 157 (内存缓存关闭: `--direct=1`, 每项前 drop 客户端+服务端 page cache) |
| 存储磁盘 | 6 × 7TB NVMe XFS |

## 测试口径

| 参数 | 值 |
|------|-----|
| 模式 | cold (`--direct=1`, `drop_caches` 客户端+全部 storage server) |
| 顺序 bs | 256K |
| 顺序并发 | 单线程 + 16 线程 |
| 布局 | 128 jobs × 1G = 128G, bs=4M |
| 随机 bs | 256K |
| 随机并发 | 128 jobs, ioengine=libaio, iodepth=128, runtime=60s, 3 轮 |
| Block size sweep | randread bs=64K/256K/1M, 3 轮 |

## 限速方案

通过 `connInterfacesFile` 强制 BeeGFS 走独立网卡 `eno12409`，在该网卡上加 `tc tbf rate 1gbit` 模拟千兆环境。
157 上的 K8s 业务走 `eno12399` + `tunl0`，不受影响。

限速命令：`bash limit-bandwidth.sh apply / remove`

## 测试流程

1. 清空历史数据 + 全部缓存
2. apply 限速
3. **限速 Round 1** → `full-cold-limit-r1-20260703-195538/`
4. 清数据 + 清缓存
5. **限速 Round 2** → `full-cold-limit-r2-20260703-215252/`
6. 清数据 + 清缓存
7. remove 限速
8. **不限速 Round 1** → `full-cold-nolimit-r1-20260703-232434/`
9. 清数据 + 清缓存
10. **不限速 Round 2** → `full-cold-nolimit-r2-20260703-235256/`

## 分析

1. **BeeGFS 限速 vs JF+Ceph 二轮调优**：单线程读/写接近（BeeGFS +12%~16%），多线程和随机 I/O 差距大（2.8x~9.1x），主要因为 BeeGFS 3 节点 NVMe stripe 并行 vs Ceph HDD EC 4+2 的单 OSD 瓶颈。
2. **BeeGFS 限速 randread 339 MiB/s > 1Gbps 线速**：限速在 3 个 slave egress 各 1Gbps，客户端并行拉取聚合 ≈ 3×1Gbps。单客户端千兆场景需在客户端也限速。
3. **不限速 randread 10.6 GB/s**：BeeGFS stripe 并行从 3 节点 × 100GbE 拉取，6 NVMe 聚合读。
4. **限速两轮偏差 <1%**：服务端清缓存 + `drop_all_caches` 有效消除了 XFS page cache 影响。
5. **seqwrite 限速 58.9 vs 不限速 335（5.7x）**：Buddy Mirror 复制消耗写带宽，限速后瓶颈在客户端 egress。
