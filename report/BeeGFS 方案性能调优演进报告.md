# BeeGFS 方案性能调优演进报告

> 环境：4 节点 BeeGFS 7.3.2（157 mgmtd+meta+client / 150·151·152 meta+2storage）/ Buddy Mirror（meta 2 group + storage 3 group，chunk=1M，3 targets）/ 6 × 7TB NVMe XFS / 数据面 100GbE RDMA(RoCE)
> 对比口径：限速 1Gbps（TBF eno12409）对齐 JuiceFS+Ceph 千兆场景；不限速暴露后端真实能力
> 达标标准：千兆场景单项带宽 ≥ 59 MB/s（与 JuiceFS 报告一致）
> 数据来源：全量 fio 冷态测试（direct=1 + drop 客户端+3 slave page cache），原始日志可追溯至 `results/<日期>-*/`

## 二、关键调优突破

### 突破 1：基线可信度修复（阶段 0b）

调优起步时发现 07-03 初始基线存在 5 类可信度问题，必须先修复否则后续对比全部失真：

| 问题 | 修复 | 影响 |
|------|------|------|
| meta 节点4（152）通信错误，target 4 容量 0 | 服务自然恢复，4 meta + 6 storage 全 GOOD | 镜像冗余完整 |
| `beegfs-ctl --mgmtd_node` 在 7.x 失效，拓扑采集全失败 | 去 `--mgmtd_node`，依赖 client.conf | env 快照可用 |
| `iopsget()` 大写匹配致 IOPS 全 NA | 改小写 `read:`/`write:` + 支持 k/M 后缀 | summary IOPS 可信 |
| `bench-basic.sh:70-71` 递归栈溢出 | 改内联 drop 逻辑 | 脚本可跑 |
| seqread R1=565 vs R2=1521（+170%）超 10% 噪声线 | 5 轮重跑均 1489-1623，偏差 9% | 认定为"部署后首次冷启动"一次性效应（RDMA 连接/元数据首建），565 不可复现 |

修复后建立的 v2 基线是后续所有对比的锚点。

### 突破 2：数据面认知纠正 —— 100GbE RDMA 而非 10GbE TCP（阶段 0b，核心突破）

原始计划（`01-bottleneck-and-tuning-plan.md` §2.6）误判 BeeGFS 走 10GbE 管理网，并据此推算"单流写 clat 708µs 中 ~60% 是 10GbE 每 chunk 串行传输"，规划了"切 100GbE"整个阶段。

实测推翻：`connUseRDMA=true`；`ibstat` 两端口 100Gb ACTIVE（4X EDR / RoCE）；randwrite 写 366GiB 而 `/proc/net/dev` TX 仅增 227MB（0.06%）—— 流量绕过 TCP 栈走 RDMA。v2 单流写 clat 已是 263µs（RDMA），非 10GbE 的 708µs。

**影响**：原计划 §2.6 整段作废，"切 100GbE"阶段删除。集群本就在 100GbE RDMA 上，网络层无调优空间（且受共用约束不可动，见突破 3）。

### 突破 3：共用网卡安全红线确立（阶段 0b）

`fuser /dev/infiniband/uverbs0` 显示同一 RDMA verbs 设备上同时挂着 `beegfs-meta` 与十几个 `wekanode`（WekaIO 生产业务）。100GbE Mellanox 网卡被 BeeGFS 与 WekaIO 物理共用。

由此确立贯穿后续所有阶段的调优边界：

| 层级 | 是否可动 |
|------|---------|
| BeeGFS 应用层缓冲（connRDMABuf*、tuneNumWorkers）| ✅ 可调（业务低峰重启） |
| slave（150/151/152）内核参数 | ✅ 可调 |
| 子目录 stripe（`--setpattern`）| ✅ 可调（不动根目录） |
| RoCE QoS / DSCP / PFC | ⚠️ 默认不动（与 WekaIO 共享） |
| 网卡/驱动全局（MTU、mlxconfig、中断亲和）| ❌ 禁止 |
| 157 内核参数 | ❌ 禁止（保护 K8s+WekaIO） |

这条红线使 BeeGFS 的可调空间显著窄于 JuiceFS（后者可自由调客户端/缓存/网络）。

### 突破 4：归因结论的自我推翻（阶段 1 前置，诚实记录）

阶段 0c 的对照实验曾得出"slave 调优对单流写 +78%（466→835），主因 dirty_ratio 20→10"，并被写入 v2 交付文档。阶段 1 前置复测独立重跑 v2 基线，发现 **835 不可复现**：

| 测量时刻 | 事件 | seqwrite | clat min |
|------|------|:---:|:---:|
| 07-06 18:44 | v2 基线（0c 对照实验的"tuned"引用源） | 835 | 200µs |
| **07-06 20:23** | **BeeGFS 服务重启** | - | - |
| 07-07 11:47 | 对照实验 untuned | 466/469 | ~400µs |
| 07-07 12:48/13:01 | 复测 R1/R2（tuned） | 479/479 | 400µs |
| 07-07 14:05 | 空闲态实测（WekaIO RDMA 流量=0） | 508 | 403µs |

排查链：
- 排除 slave 参数漂移、RDMA 降速/错误、slave NVMe 变慢、CPU 绑核、**WekaIO 业务争抢（空闲态仍 508）**
- **关键验证**：dirty_ratio 10 vs 20 对 `--direct=1` 几乎无差异（512 vs 503）—— direct write 不经 page cache，dirty_ratio 本就无效
- 真相：对照实验的"tuned=835"（重启前）与"untuned=466"（重启后）跨越了**服务重启这个未受控变量**；466→835 的差异全部来自服务重启，不是 dirty_ratio

**修正**：
- ❌ 撤回"slave 调优对单流写 +78%"
- ❌ 撤回"v2 单流写 835 为基线锚点"（n=1、无快照、不可复现孤例）
- ✅ 当前真实单流写基线 ≈ **479-512 MiB/s**
- ✅ layout +2.5%、100GbE RDMA、共用网卡红线、写天花板/小块读瓶颈等其余结论不受影响

**遗留未解**：为什么重启前 835（clat min 200µs）、重启后稳定 ~490（clat min 400µs），机制未知。clat 增量 ≈ 200µs 来自 BeeGFS 协议层/RDMA 传输层，但 RDMA 无错误、NVMe 延迟正常。已派发重启可复现性实验（≥8 轮重启+快照对照）定位。

### 突破 5：千兆限速基线稳定可对标（贯穿全阶段）

与 JuiceFS 报告同口径的 1Gbps 限速场景，BeeGFS 数据高度稳定（07-03 与 v2 两轮偏差均 <1%），是两方案公平对比的可靠锚点：

| 测试 | BeeGFS 限速 (MiB/s) | JF+Ceph 冷态最优 (MiB/s) | BeeGFS 优势 |
|------|:---:|:---:|:---:|
| 顺序读 | 114 | 80.1 | +42% |
| 顺序写 | 58.9 | 57.0 | +3% |
| 多线程读 | 302 | 109 | +177% |
| 多线程写 | 113 | 43.7 | +159% |
| 布局写 | 113 | 39.1 | +189% |
| 随机读 | 339 | 52.0 | +552% |
| 随机写 | 113 | 55.7 | +103% |
| 随机读写 R/W | 109/108 | 26.6/26.3 | +310%/+311% |

> 限速在 3 个 slave egress 各 1Gbps，客户端并行拉取聚合 ≈ 3×1Gbps，故 randread 339 > 单千兆线速（~124 MB/s）。单客户端千兆场景需补客户端侧限速（阶段 4 任务）。

## 三、深度诊断工作

### 3.1 瓶颈分层定位（v2 不限速基线）

| 瓶颈层 | 表现 | 定位 |
|--------|------|------|
| 读天花板 ≈ 8.2-8.8 GB/s | randread 8227 / randread-1M 8807 | 读只命中 primary target 无镜像开销，6 NVMe × 100GbE RDMA 聚合，接近硬件上限。**读侧无明显瓶颈** |
| 写天花板 ≈ 6 GB/s | randwrite 6138 / randrw 4602+4599 / layout 10240 收敛 | Buddy Mirror 每 chunk 同写 primary+secondary，**写放大 2×**，需两端落盘确认。物理写 ≈ 12 GB/s，接近 6 NVMe 聚合。瓶颈 = 镜像同步复制 + NVMe 落盘 |
| 单流写最突出短板 | seqwrite 479-512 vs randwrite 6138（12×） | clat 485µs，IOPS ~1900。**延迟主导非带宽**：write→primary→secondary（RDMA 双写同步）+ 2× NVMe 落盘 ack 串行 |
| 小块随机读退化 | randread-64K 4789 vs 256K 8227 vs 1M 8807 | per-IO meta/网络往返固定开销占比高 |

### 3.2 seqread R1≠R2 根因分析

07-03 现象：单流 seqread 同一次运行内 R1=565 → R2=1521（+170%），而多流读 R1/R2 仅差 2%。

5 轮冷态重跑 + 暖态对照：5 轮 cold 均为 1489-1623（偏差 9% < 10%），**从未复现 565**。根因判定为"全新部署后首次 benchmark 的一次性冷启动效应"：clean-beegfs.sh + deploy-beegfs.sh 全部服务重启后，客户端 RDMA 连接冷态 + 元数据缓存空，R1 需建连接+拉元数据。drop_all_caches 清不掉 RDMA 连接和元数据缓存，故冷态/暖态无法区分。系统运行 3 天后不再复现。

### 3.3 已排除的无效方向

| 方向 | 排除依据 |
|------|------|
| 切 100GbE（原计划阶段2） | 实测本就是 100GbE RDMA，前提错误 |
| slave dirty_ratio 精调（原阶段1 变量A） | direct=1 不经 page cache，10 vs 20 实测无差异（512 vs 503） |
| 157 共部署争抢 | seqwrite/randwrite 期间 BeeGFS 服务 0% CPU；96 核充裕；空闲态（WekaIO RDMA 流量=0）seqwrite 仍 508，非争抢 |
| WekaIO 业务争抢致 seqwrite 降速 | 空闲态实测 508 vs 繁忙态 479 同量级；两轮 479=479 完全一致（争抢会波动），clat min 稳定翻倍 → 确定性状态变化 |
| RDMA 链路降速/错误 | port state 100Gb ACTIVE，port_xmit_discards/wait/rcv_errors 全 0 |
| slave NVMe 变慢 | iostat w_await 0.18-0.21ms 正常 |
| CPU 调度竞争 | taskset 绑核 0-3 后 504，无改善 |
| 07-03 layout=1640 作为对比锚点 | 对照实验证 slave 调优对 layout 仅 +2.5%，未调优对照也有 9964 → 1640 是首日单轮污染值 |

### 3.4 当前未解决问题

| 问题 | 状态 |
|------|------|
| 服务重启前 835 / 重启后 ~490 的机制 | **未定**。clat min 翻倍（200→400µs）来自 BeeGFS 协议/RDMA 层，但 RDMA 无错误。可能：RDMA 连接重建走次优路径 / storage 运行时状态变化 / 服务运行时长影响 RDMA buffer 回收。已派发重启可复现性实验定位 |
| 单流写如何从 ~490 继续提升 | **阻塞于上一项**。在不可信基线上做单变量调优无法得出有效结论。stage1 调优暂缓 |
| randread 复测偏高（+12%）/ randread-1M 偏高（+20%） | 推测 WekaIO 持续 RDMA 流量预热 PCIe/RDMA 路径降低读延迟。读项偏高不影响写调优，但说明 v2 读项也可能不可信 |

### 5.2 不限速后端能力（v2 基线，部分待复测确认）

| 指标 | v2 值 | 复测状态 | 说明 |
|------|-------|---------|------|
| 单流读 | 1585 | 稳定（1484-1623） | RDMA 100GbE 接近硬件上限 |

### 5.3 残余瓶颈与下一步

> 不限速后端能力的瓶颈定位：

- **单流写是最大短板**（479-512 vs 并发写 6138，12× 差距）：延迟主导非带宽，写路径 client→primary→secondary 双写同步 + 2× NVMe 落盘 ack 串行。是离硬件上限最远、最值得调的方向，但**当前阻塞于基线可信度**。
- **写天花板 ~6 GB/s 受限于 Buddy Mirror 2× 写放大**：物理写 ~12 GB/s 接近 6 NVMe 聚合。需用非镜像子目录对照量化镜像代价（阶段2）。
- **小块随机读退化**（64K 4789 vs 256K 8227）：per-IO 固定开销占比高，待 100GbE 基线上调优 read_ahead/iodepth（阶段3）。

**本周进展（2026-07-06~07）：**
- **v2 健康基线建立**：修复 5 类可信度问题，拿到可复现的全矩阵数据（限速偏差 <1%）。
- **两处核心认知纠正**：数据面是 100GbE RDMA（非 10GbE，原"切 100GbE"阶段作废）；100GbE 网卡与 WekaIO 共用（确立安全红线）。
- **slave 调优 +78% 结论自我推翻**：复测发现 835 不可复现，差异来自服务重启非 dirty_ratio；dirty_ratio 对 direct=1 实测无效。当前真实单流写基线 ~479-512。
- **千兆限速基线全部达标**：与 JuiceFS 同口径对比，BeeGFS 各项均达标且显著优于 JF+Ceph。
- **阻塞项**：单流写"重启前 835 / 重启后 490"机制未定，已派发重启可复现性实验（≥8 轮重启+快照对照）。在其出结论前 stage1 单变量调优暂缓，基线暂以 ~490 计。

**下一步路径**：
1. 重启可复现性实验出结论（A/B/C 三态）→ 确立 stage1 真实基线
2. stage1 单变量矩阵（connRDMABuf*/tuneNumWorkers/chunksize/fsync）→ 单流写从 ~490 提升
3. stage2 非镜像对照 → 量化 Buddy Mirror 2× 写放大代价
4. stage3 小块随机读优化
5. stage4 暖态基线 + 单客户端千兆限速 + 与 JuiceFS 终版同口径对比

## 六、与 JuiceFS 方案的对比小结

| 维度 | BeeGFS | JuiceFS+Ceph |
|------|--------|--------------|
| 数据面 | 100GbE RDMA（无网络调优空间，受共用约束） | 1Gbps TCP（已修复 100Mb→1G，无 RDMA） |
| 后端 | 6 NVMe + Buddy Mirror 2× 写放大 | Ceph EC 4+2 + RGW/librados |
| 千兆达标 | **全部达标**（限速基线，无须进一步调优） | 部分达标（顺序写差3%、随机读写差55%） |
| 调优空间 | 受共用网卡红线限制，仅 BeeGFS 应用层/slave 端可动 | 客户端/缓存/网络均可调 |
| 主要瓶颈 | 单流写延迟（服务重启机制未定，阻塞 stage1） | EC 写放大 + 1Gbps 带宽 + loadRange bug（已定位修复） |
| 调优成熟度 | 阶段0 完成，stage1 阻塞中（基线可信度优先） | 阶段4 完成，残余瓶颈 randrw 差 55% |
| 核心方法论 | 基线可信度优先 + 单变量对照 + 结论可自我推翻 | 白盒实验 + 源码分析 + patch 验证 |

> 两方案调优路径迥异：JuiceFS 在稳定基线上持续参数优化，最终定位并修复了 v1.3.1 loadRange bug；BeeGFS 则在调优起步阶段投入大量精力建立可信基线，并诚实推翻了自身的归因结论。当前 BeeGFS 千兆场景已全面达标，不限速后端能力的进一步调优待基线阻塞解除后推进。
